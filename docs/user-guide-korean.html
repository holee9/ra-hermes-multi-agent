시스템 개요

현재 상태: 모든 컴포넌트가 구축 완료되어 실제 운영 중입니다. ✅ 작동 중

아키텍처 개요

RA Hermes 시스템은 Hermes Agent v0.15.1과 Honcho v0.15.1을 기반으로 구축된 학습형 멀티 에이전트 시스템입니다. 8종의 AI 에이전트가 실제 업무를 보조하며, 지속적인 피드백을 통해 성장합니다.

주요 특징

  • T자형 성장: 공통 지식 토대(llm-wiki, ra-project) + 개별 전문성(Honcho 메모리)
  • 정확성·신뢰성 우선: 의료기기 규제에서 오류는 환자 안전 문제 → 불확실 시 사람에게 올림
  • 학습하며 성장: 실제 업무 경험과 피드백으로 전문성이 깊어짐
  • 분산 의사결정: 인프라 3종 투표 시스템 (단일 장애점 없음)

장비 배치

장비 사양 역할 상태

🏢 가상 오피스 사용법

현재 상태: 실제 Honcho 활동 기록을 시각화하여 재생 중입니다. ✅ 작동 중

시스템 개요

가상 오피스는 RA Hermes 시스템의 활동 이벤트를 픽셀 아트 그래픽으로 시각화하는 관찰자 시스템입니다. 실제 Honcho 데이터를 기반으로 에이전트들의 활동을 재생합니다.

주요 기능

  • 활동 이벤트 시각화: mail_received, matched, comment_added 등 7종 이벤트
  • 캐릭터-에이전트 매핑: 8종 RA/인프라 에이전트를 픽셀 캐릭터로 표현
  • 실시간 재생: Honcho 활동 기록을 실시간으로 재생
  • 속도 조절: 정지/재생/1배속/2배속 4가지 모드

사용법

  1. 브라우저에서 virtual-office/virtual-office.html 열기
  2. 상단 컨트롤에서 속도와 재생 기능 조작
  3. 캐릭터 클릭으로 에이전트 정보 확인 가능

에이전트 오피스 내 위치 역할

📊 성장 모니터링 사용법

현재 상태: GitHub Pages에서 실제 성장 데이터를 기반으로 대시보드 배포 중. ✅ 배포 중

대시보드 접속

성장 대시보드 바로가기

주요 기능

  • RA Growth Operations 요약: 전체 시스템 성장 현황 한눈에 파악
  • 담당자별 성장 카드: ra_us/ra_eu/ra_kr 개별 성장 추적 및 지표
  • Growth Signal Flow: 성장 신호 흐름 시각화
  • 성장 측정 상태: Trend Verdict, Evidence Radar 차트로 분석
  • 커버리지 근거: coverage-guards.json 기반 판정

지표 설명

지표 설명 데이터 소스
correction_rate 사람 수정 비율 (낮을수록 좋음) feedback 기록
first_pass_match_accuracy 첫 매칭 정확도 WP 매칭 기록
confidence_calibration 신뢰도 보정 여부 RA 응답 confidence
warmstart_lift Honcho 컨텍스트 활용 효과 컨텍스트 사용 전후 비교

대시보드 해석 방법

성장 카드 읽는 법
  • 상단 카테고리: 신호 강도 (강함/중간/약함)
  • 좌/하단: 최근 7일/30일 트렌드 방향
  • 지표: 위 5개 지표의 현재 값과 추세

→ 카드가 "녹색"이면 성장 중, "주홍"이면 성장 둔화 중

Growth Signal Flow 읽는 법
  • 메일 → RA 분석 → WP 매칭: 핵심 루프의 흐름
  • 피드백 → 학습: 3점 평가가 지식베이스에 반영
  • 자율 학습 → 전문성 심화: Bootstrap/Delta 학습 루프

→ 흐름이 연속되면 정상, 막히면 어딘가 문제

주의: 현재 threshold/notification는 30일 valid metrics 전까지 null 유지되어 자동 알림이 비활성화되어 있습니다.

⚙️ n8n 워크플로우 사용법

현재 상태: 4개 핵심 워크플로우가 T3610 n8n에서 활성화되어 운영 중. ✅ 4개 활성

n8n 접속

T3610 n8n: http://T3610:5678 (내부망)

주요 워크플로우

1. mail-triage.json - 메일 트리아지

기능

  • Gmail 트리거 → 재전송 본문 파싱
  • 규제권 라우팅 (US/EU/KR)
  • RA 에이전트 분석 호출
  • OpenProject WP 자동 생성/코멘트

Yellow 게이트 (안전장치)

  • low confidence: 신뢰도 낮음 → 사람 검토
  • 완료 WP 매칭: 종결된 WP → 사람 확인 필요
  • OpenProject 조회 실패: 연결 문제 → 사람 검토
2. infra-vote-broadcast.json - 인프라 투표

기능

  • 인프라 3종(Finn/Leo/Gus) 투표 소집
  • 투표 결과 집계 (quorum 2/3 majority)
  • 업무 workspace로 결과 전달

투표 규칙 (초기값)

  • 정족수: 2/3 (과반수)
  • 다수결: 66% 이상 찬성
  • 기권: 0, 반대 1, 기권 2
3. feedback-recorder.json - 피드백 기록

기능

  • 사람 3점 평가 (나쁨/보통/좋음) 수신
  • Honcho에 평가 기록
  • 가중치 조정 (env/config 외부화)
  • 차회 학습에 반영

사용법

  1. 피드백 Webhook URL 설정
  2. 평가 요청: {"agent_judgment": "bad", "human_correction": "good", ...}
  3. 기록 확인: Honcho 세션 기록에서 확인
4. infra-to-work-bridge.json - 인프라→업무 브릿지

기능

  • 인프라 workspace 합의 결론 전달
  • 단방향 흐름 (인프라→업무)
  • 업무 workspace에 기록

안전 장치

  • Honcho 자체 장애 시 우회 경로
  • n8n이 우회하여 직접 경보
  • 순환 의존 차단
5. form-triage-draft.json - 폼 입력 처리

기능

  • 폼 입력 → mail-triage 계약으로 변환
  • RA 분석 재사용
  • 폼 데이터 검증

현재 상태

운영 보류: FORM_TRIAGE_ENABLED=false (30일 valid metrics 미달)

→ 폼 기반 처리는 운영 데이터 충족 후 활성화 예정

환경변수 설정

중요: 모든 워크플로우는 n8n/.env와 n8n/docker-compose.yml에서 환경변수를 관리합니다.
환경변수 필수 여부 용도
HONCHO_API_URL 필수 Honcho API base URL
OPENPROJECT_API_URL 필수 OpenProject API base URL
YELLOW_CONFIDENCE_THRESHOLD 필수 Yellow 전환 confidence 기준
HUMAN_ALERT_WEBHOOK_URL 선택 Yellow payload 전송 Webhook
